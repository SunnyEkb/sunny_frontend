# Архитектура проекта Sunny Frontend

Этот документ описывает структуру и архитектуру фронтенд-приложения Sunny.

## Корневая структура

Ниже приведено описание основных файлов и каталогов в корневой директории проекта.

- **`.cursorrules`**: Правила конфигурации для среды разработки Cursor.
- **`.editorconfig`**: Обеспечивает единый стиль кодирования в различных редакторах и IDE.
- **`.eslintrc.cjs`**: Файл конфигурации для ESLint, инструмента для поиска и исправления проблем в коде JavaScript/TypeScript.
- **`.gitignore`**: Содержит список файлов и папок, которые система контроля версий Git должна игнорировать.
- **`Dockerfile`**: Инструкции для сборки Docker-образа приложения. Позволяет запускать приложение в изолированном контейнере.
- **`GEMINI.md`**: Файл с инструкциями и контекстом для AI-ассистента Gemini.
- **`index.html`**: Основная HTML-страница, которая служит точкой входа для всего веб-приложения.
- **`package.json`**: Определяет метаданные проекта, зависимости (библиотеки) и скрипты для сборки, тестирования и запуска.
- **`package-lock.json`**: Фиксирует точные версии всех установленных зависимостей для обеспечения консистентности окружения.
- **`README.md`**: Вводный файл с общей информацией о проекте, инструкциями по установке и использованию.
- **`tsconfig.json` / `tsconfig.node.json`**: Файлы конфигурации для компилятора TypeScript.
- **`vite.config.ts`**: Файл конфигурации для Vite, современного сборщика фронтенд-проектов.

### Каталоги

- **`.git/`**: Скрытая папка, содержащая всю историю и метаданные Git-репозитория.
- **`.github/`**: Содержит файлы, связанные с GitHub, в основном для автоматизации процессов (CI/CD) через GitHub Actions.
- **`dist/`**: Папка, в которую помещаются скомпилированные и оптимизированные файлы после сборки проекта. Содержимое этой папки развертывается на сервере.
- **`node_modules/`**: Хранит все зависимости проекта (библиотеки и пакеты), загруженные через npm или yarn.
- **`public/`**: Содержит статические активы, которые не требуют обработки сборщиком и доступны напрямую по URL.
- **`src/`**: Основная папка с исходным кодом приложения.

## Исходный код (`src/`)

### `src/app/`

Этот каталог отвечает за основную логику приложения, включая маршрутизацию, основные макеты страниц и защиту роутов.

- **`MobileOnlyRoute.tsx`**: Компонент-обертка для роутов, которые должны быть доступны только на мобильных устройствах.
- **`moderatorProtectedRoute.tsx`**: Защищенный роут, доступ к которому имеют только пользователи с правами модератора.
- **`paths.ts`**: Файл, который экспортирует константы с путями страниц приложения. Это помогает избежать ошибок и упрощает рефакторинг.
- **`protectedRoute.tsx`**: Защищенный роут, доступный только для аутентифицированных пользователей.
- **`router.tsx`**: Центральный файл конфигурации маршрутизации. Здесь определяется, какой компонент будет отображаться для каждого URL-адреса.
- **`layouts/`**: Каталог с компонентами-макетами страниц (например, `MainLayout` с шапкой и подвалом).

### `src/app/layouts/`

Этот каталог содержит компоненты, отвечающие за структуру и макеты страниц приложения.

-   **`MainLayout.tsx`**: Основной компонент макета, который оборачивает контент страниц. Он включает в себя общие элементы, такие как шапка (`Header`) и подвал (`PcFooter`, `FooterButtonMain`). `MainLayout` также может скрывать эти элементы на определенных страницах (например, на странице модерации) в зависимости от текущего URL. Стили для этого компонента находятся в `MainLayout.module.scss`.

-   **`RootLayout/RootLayout.tsx`**: Корневой макет приложения, который управляет отображением других макетов и модальных окон. Он использует `react-router-dom` для рендеринга дочерних маршрутов (`<Outlet />`). В зависимости от текущего пути (`location.pathname`), `RootLayout` может применять разные макеты. Например, для страницы профиля (`/profile`) он отображает контент напрямую, а для остальных страниц оборачивает его в `SearchProvider` и `MainLayout`. Кроме того, `RootLayout` отвечает за отображение модального окна аутентификации (`AuthModal`), когда оно активно.

-   **`RootModal/RootModal.tsx`**: Универсальный компонент модального окна, который рендерится поверх остального контента с помощью React Portal (`createPortal`). Это обеспечивает его корректное отображение и изоляцию от основного DOM-дерева. Компонент может принимать пропсы для настройки, например, `backdropFilter` для добавления эффекта размытия фона. Стили модального окна определены в `rootModalStyle.module.scss`.

-   **`SearchProvider/SearchProvider.tsx`**: Провайдер контекста для функциональности поиска. Он хранит состояние строки поиска (`searchString`) и предоставляет функцию (`handleSearchItems`) для ее обновления. Хук `useSearch` экспортируется для удобного доступа к состоянию и функции поиска из любого дочернего компонента. Это позволяет компонентам, таким как поисковая строка в шапке и страница с результатами, обмениваться данными без необходимости передавать их через пропсы.

### `src/ads/`

Этот каталог представляет собой основной модуль, отвечающий за все, что связано с "объявлениями" и "услугами" в приложении. Он инкапсулирует в себе полный цикл жизненного цикла объявлений: от создания и модерации до отображения и взаимодействия с ними.

Структура каталога разделена на две основные части: `components` и `pages`.

-   **`components/`**: Содержит переиспользуемые UI-компоненты, которые являются строительными блоками для страниц.
    -   `search/`: Компоненты, связанные с функциональностью поиска.
    -   `widgets/`: Различные виджеты, например, для отображения услуг (`widgetUslugi.tsx`).

-   **`pages/`**: Каждый подкаталог здесь представляет собой отдельную страницу (или сложный компонент, занимающий основную часть страницы), доступную по определенному маршруту.
    -   **`createAds/`**: Логика и компоненты для создания новых объявлений и услуг. Представляет собой многошаговую форму, где пользователь выбирает тип и категорию, заполняет информацию, загружает фотографии и отправляет на модерацию.
    -   **`catalog/`** и **`catalogs/`**: Страницы для просмотра списков объявлений и услуг. `catalogs` является, по-видимому, страницей верхнего уровня с категориями, в то время как `catalog` отображает отфильтрованный список с возможностью поиска и фильтрации.
    -   **`CardCatalogBig/`**: Страница детального просмотра одного объявления или услуги. Здесь отображается вся информация: изображения, описание, цена, автор, а также реализованы функции добавления в избранное, "поделиться" и секция с комментариями.
    -   **`typeCatalog/`**: Страница для выбора конкретного типа или подкатегории услуг для последующей фильтрации.
    -   **`moderation/`**: Панель для модераторов. Предоставляет интерфейс для просмотра и управления объявлениями, услугами и комментариями, ожидающими проверки. Это ключевая часть системы для обеспечения качества контента.
    -   **`chatPage/`**: Реализация чата между пользователями, скорее всего, между покупателем и продавцом в контексте конкретного объявления. Использует WebSockets для обмена сообщениями в реальном времени.
    -   `prodazja/` и `uslugi/`: Вероятно, страницы, специфичные для продажи товаров и услуг соответственно.

В целом, модуль `ads` является хорошо структурированным и самодостаточным, реализуя значительную часть бизнес-логики приложения.

## src/store

Директория `src/store` отвечает за управление состоянием приложения с использованием Redux Toolkit. Она структурирована по функциональным модулям и следует лучшим практикам Redux.

### Основные файлы и директории:

-   **`store.ts`**:
    -   Главный файл, конфигурирующий Redux store.
    -   Использует `configureStore` для создания стора.
    -   Объединяет все редьюсеры (reducers) из различных slices и RTK Query API.
    -   Подключает мидлвары (middleware), включая мидлвары от RTK Query для обработки асинхронных запросов и кастомный мидлвар для WebSocket.
    -   Экспортирует типы `RootState` и `AppDispatch` для использования во всем приложении, а также типизированные хуки `useAppDispatch` и `useAppSelector`.

-   **`actions/`**:
    -   Содержит файлы с определениями действий (actions) для Redux.
    -   **`chat.ts`**: Определяет экшены для управления WebSocket-соединением чата (подключение, отключение, отправка и получение сообщений).

-   **`auth-api/`**:
    -   Содержит RTK Query API для взаимодействия с эндпоинтами, связанными с аутентификацией.
    -   **`authApi.ts`**: Определяет эндпоинты для логина, регистрации, выхода, проверки аутентификации, восстановления пароля и т.д. Автоматически управляет кэшированием и состоянием запросов.

-   **`entities/`**:
    -   Эта директория, по всей видимости, содержит RTK Query API slices для различных сущностей данных в приложении (объявления, комментарии, уведомления и т.д.), что соответствует импортам в `store.ts`.

-   **`midlewares/`**:
    -   Содержит кастомные мидлвары для Redux.
    -   **`MidlewaresWebSoket.ts`**: Реализует мидлвар для работы с WebSocket (`socket.io-client`). Он перехватывает экшены, связанные с чатом, и управляет жизненным циклом WebSocket-соединения.

-   **`slices/`**:
    -   Содержит "слайсы" (slices) состояния, созданные с помощью `createSlice` из Redux Toolkit.
    -   **`authSlice.ts`**: Управляет состоянием аутентификации пользователя (`isAuthenticated`, `user`). Содержит редьюсеры для установки и сброса данных пользователя. Реагирует на успешное/неуспешное выполнение запроса `checkAuth` из `authApi` с помощью `extraReducers`.
    -   **`serviceSlice.ts`**: Небольшой слайс для управления состоянием UI, связанного с сервисами (например, пагинация и поиск).
    -   **`wsChatSlice.ts`**: Управляет состоянием чата, включая список сообщений и статус соединения. Обновляется в ответ на экшены, диспатченные WebSocket-мидлваром.

### Общая архитектура:

Архитектура `store` построена на основе Redux Toolkit, что позволяет эффективно управлять состоянием и асинхронными операциями. RTK Query используется для декларативного описания API-взаимодействий, что упрощает работу с данными и их кэширование. Для реалтаймового общения (чат) используется кастомный WebSocket мидлвар, который интегрирован в Redux-цикл. Разделение на `slices` и `api` по функциональному признаку делает код модульным и легко расширяемым.

### `src/user/`

Этот каталог содержит все, что связано с пользователем: его профиль, аутентификация, настройки и другие связанные страницы и компоненты. Модуль инкапсулирует в себе всю логику взаимодействия пользователя с его личными данными.

-   **`components/`**: Здесь находятся переиспользуемые компоненты, специфичные для пользовательского раздела. Например, компоненты для аватара, формы, футера, хедера, инпутов и т.д.
-   **`layout/`**: Компоненты-макеты для страниц личного кабинета пользователя. Например, `authPageLayout` для страниц аутентификации и `layoutUserLK` для страниц личного кабинета.
-   **`pages/`**: Содержит компоненты, которые представляют собой полноценные страницы, доступные по определенным маршрутам. Сюда входят страницы для входа (`logIn`, `singin`), регистрации (`registerActivate`), восстановления пароля (`passwordRecovery`, `newPassword`), настроек пользователя (`settingsUser`), просмотра объявлений пользователя (`UserAds`), избранного (`UserFavorites`), сообщений (`UserMessages`), отзывов (`UserReviews`) и редактирования профиля (`UserProfileEdit`).
-   **`providers/`**: Провайдеры контекста для пользовательских данных.
