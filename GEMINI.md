# Контекст и правила Gemini: sunny_frontend

Этот файл содержит архитектурный контекст, технические правила и поведенческие гайдлайны для ИИ-агента, работающего над проектом `sunny_frontend` (Платформа/Доска объявлений). Всегда строго следуй этим инструкциям.

## Обязательное требование: все ответы и рассуждения всегда должны быть только на русском языке.

---

## 🛠 Технологический стек
* **Основа:** React 18, TypeScript 5.
* **Сборщик:** Vite 5 (ESModules).
* **Стейт-менеджер:** Redux Toolkit (хранится в `src/store`).
* **Роутинг:** React Router DOM v7 (`src/app/router.tsx`).
* **Валидация форм:** React Hook Form + Yup.
* **Сеть:** Axios + Socket.io-client (для реалтайм-чата `chatPage`).
* **Стили:** SASS/SCSS (`.module.scss`), FontAwesome v6, Swiper.

## 📂 Архитектурная структура
Проект использует модульно-слоистую структуру. Код должен строго разделяться по назначению:
1. `src/app/` — Инициализация приложения, глобальные провайдеры, роутинг (`router.tsx`), защищенные маршруты (`protectedRoute.tsx`) и лэйауты (`layouts/MainLayout.tsx`).
2. `src/ads/` — Продуктовый модуль объявлений.
   * `ads/pages/` — Страницы (каталоги, создание объявлений `createAds`, чат `chatPage`, модерация).
   * `ads/components/` — Специфичные для объявлений компоненты, виджеты и поиск.
3. `src/user/` — Продуктовый модуль пользователей (профили, авторизация).
4. `src/shared/` — Переиспользуемые независимые элементы (UI-компоненты: `button`, `Modals`, `notification`; общие инструменты: `api`, `hooks`).
5. `src/common/` — Глобальные типы/модели данных (`model/ads.ts`) и общие SASS-переменные (`variables/variables.scss`).

## 📄 Дополнительные материалы
* Всегда учитывай информацию из файла `PROJECT_ARCHITECTURE.md` при работе над задачами.

## 🎯 Технические правила разработки

### 1. Компоненты и Стили
* Пиши только **функциональные компоненты** (стрелочные функции) с хуками.
* Для стилей используй **CSS-модули SASS** (например, `ComponentName.module.scss`).
* Импортируй глобальные SCSS-переменные из `src/common/variables/variables.scss`.
* Для SVG-иконок из `src/assets/icon/` используй `vite-plugin-svgr` (импорт как React-компонентов).

### 2. Типизация (TypeScript)
* Никаких `any`. Используй строгую типизацию для всех props, состояний и ответов API.
* Общие сущности (например, интерфейс объявления `Ad`) бери или добавляй в `src/common/model/`.

### 3. Маршрутизация и Доступ
* Новые страницы регистрируй в `src/app/paths.ts` и `src/app/router.tsx`.
* Для приватных зон оборачивай компоненты в Guard-компоненты из `src/app/` (`protectedRoute` / `moderatorProtectedRoute`).

### 4. Работа с API и Real-time
* Сетевые запросы выноси в файлы `*Api.ts` внутри модулей или в `src/shared/api/`.
* При работе с Socket.io внутри `chatPage` или хуков **обязательно** очищай слушатели событий в `useEffect` для предотвращения утечек памяти.

## 🤖 Поведенческие правила для ИИ (Гайдлайн Claude)

Эти правила приоритизируют осторожность и точность, а не скорость.

### 1. Сначала думай, потом пиши код
* **Не предполагай и не скрывай неясности.** Открыто говори компромиссах.
* Перед реализацией четко сформулируй свои предположения. Если не уверен — спроси.
* Если есть несколько вариантов решения, предложи их, а не выбирай один молча.
* Если есть более простое решение — предложи его. Оспаривай усложнения, когда это оправдано.
* Если что-то непонятно — остановись, назови проблему и задай вопрос.

### 2. Простота в приоритете
* **Минимум кода для решения задачи. Никаких догадок и кода "на будущее".**
* Никаких фич, о которых не просили. Никаких абстракций для кода однократного использования.
* Никакой лишней "гибкости" или конфигураций, если этого не требовал пользователь.
* Никакой обработки ошибок для невозможных сценариев.
* Если ты написал 200 строк кода там, где можно было написать 50 — перепиши.
* *Задай себе вопрос:* «Скажет ли старший разработчик, что это переусложнено?» Если да — упрощай.

### 3. Точечные изменения (Хирургическое вмешательство)
* **Трогай только то, что необходимо. Убирай только свой "мусор".**
* При редактировании существующего кода не пытайся «улучшить» соседний код, комментарии или форматирование. Не рефактори то, что работает.
* Подстраивайся под существующий стиль кода, даже если сделал бы иначе.
* Если заметил несвязанный "мертвый" код — упомяни о нем, но не удаляй сам.
* Если твои изменения создали неиспользуемые импорты, переменные или функции — удали их. Прежний неиспользуемый код без спроса не трогай.
* *Критерий успеха:* Каждая измененная строка должна иметь прямую связь с запросом пользователя.

### 4. Работа на результат
* **Определяй критерии успеха. Повторяй и проверяй до полной готовности.**
* Превращай задачи в проверяемые цели (например: не просто "исправить баг", а написать сценарий/тест для воспроизведения, исправить, убедиться, что он проходит).
